# 실험 로그

## 27. PDF OCR 도입 필요성 검증

### 목적

현재 제공 RFP PDF에 기본 텍스트 추출만으로 읽히지 않는 스캔 페이지가 있는지 확인하고, OCR 도입 필요성을 결정한다.

### 검증 방법

- 대상: `/data/original_data/files`의 PDF 4개
- 기준: `pypdf`로 페이지별 텍스트를 추출했을 때 빈 페이지가 있으면 `ocr_required` 또는 `review_required`로 분류
- 재현 위치: `notebook/01_parser_chunker_test.ipynb`의 `27. PDF OCR feasibility check` 섹션

### 결과

| 파일명 | 페이지 수 | 추출 텍스트 길이 | 빈 페이지 | OCR 권고 |
| --- | ---: | ---: | ---: | --- |
| 고려대학교_차세대 포털·학사 정보시스템 구축사업.pdf | 297 | 208,445 | 0 | not_required |
| 기초과학연구원_2025년도 중이온가속기용 극저온시스템 운전 용역.pdf | 49 | 42,669 | 0 | not_required |
| 서울시립대학교_[사전공개] 학업성취도 다차원 종단분석 통합시스템 1차.pdf | 149 | 110,893 | 0 | not_required |
| 서울특별시_2024년 지도정보 플랫폼 및 전문활용 연계 시스템 고도화 용.pdf | 75 | 113,872 | 0 | not_required |

### 결론

- 현재 제공 PDF 4개는 모두 빈 페이지 없이 텍스트가 추출되어, 현 시점에 OCR을 기본 파이프라인에 도입하지 않는다.
- VM에 Tesseract OCR 실행 파일이 설치되어 있지 않으며, 스캔 PDF 후보가 없으므로 추가 설치 비용을 발생시키지 않는다.
- 이후 빈 페이지가 있는 PDF가 추가되면 `inspect_pdf_text_extraction()` 결과를 근거로 OCR 도입 여부를 다시 판단한다. 실제 OCR 파싱 구현은 이슈 47에서 진행한다.

### 한계와 후속 검토

- 이번 검증은 스캔 PDF 여부와 본문 텍스트 추출 가능성을 판단한 것이다. 표의 행·열 구조를 구조화 데이터로 복원하는 기능까지 검증하지는 않는다.
- 표 구조가 필요한 질의가 많아지면 OCR이 아니라 PDF 표 추출 또는 레이아웃 기반 파싱을 별도 실험한다.
